## Q1. 업무요청서 예제 코드를 내 방식으로 수정해도 되는가

### 상황
1주차 업무요청서에 `red.py` 예제 코드가 제시되어 있었음. 다음과 같이 수정해서 작성함:

- `subprocess`로 매 실행마다 `pip install` 하던 부분 제거
- `cv2.imshow` + `waitKey` 대신 `cv2.imwrite`로 결과 저장
- 절차형 코드를 `detect_red_pixels()` 함수로 감쌈
- 변수명을 `mask1` → `mask_low`, `lower_red1` → `lower_red_near_0` 등으로 풀어씀

### 질문
업무요청서에 제시된 예제 코드는 반드시 따라야 하는 "정답"인가, 아니면 참고용 "가이드"인가?

### 내 생각
아마 참고용 가이드일 것 같음. 업무요청서에도 "예제 코드"라고 표현되어 있고, 과제 요청 내용 자체는 "특정 색상 감지 및 필터링 기능 구현"으로 명시되어 있어서 결과만 맞으면 방식은 자유로울 듯.
다만 수정한 부분이 실무 관점에서 오히려 안 좋은 방향이진 않을지 궁금함.

## Q2. 스크립트 내 `pip install` 자동화 — 실무에서도 쓰는 패턴인가

### 상황
업무요청서 예제 코드에 `subprocess.check_call`로 매 실행 시 `pip install -r requirements.txt`를 호출하는 패턴이 있었음.
`week 1/git/red.py`와 `week 1/huggingface/image_preprocessing.py` 모두 이 방식을 따르고 있음.

### 질문
매 실행마다 `pip install`을 스크립트 안에서 호출하는 건 실무에서도 일반적인가?

### 내 생각
내가 알기론 보통 `requirements.txt`를 별도로 두고 README에 "먼저 설치하세요"라고 안내하는 방식이 더 흔한 것 같음. 매 실행마다 설치 체크하는 건 불필요한 오버헤드고, 의존성 관리도 불투명해짐.
다만 1인 개발이나 간단한 스크립트 공유 시에는 편의성 때문에 쓸 수도 있을 것 같음.
